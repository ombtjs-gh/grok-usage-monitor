# Grok 複数アカウント週間使用量モニターアプリ 設計書

**バージョン**: 1.0  
**作成日**: 2026-07-21  
**対象**: デスクトップ常駐型使用量モニター（軽量重視）

---

## 1. 概要

xAI Grok（特に SuperGrok プラン）の**週間使用量プール**を、複数アカウント分デスクトップ上に常時軽量表示するモニターアプリ。

- **対象使用量**: Settings → Usage、または Grok Build CLI の `/usage` で確認できる週間使用量プール（パーセンテージ、製品別内訳、リセット日時）
- **主要目標**: 極力軽量・低リソースで動作し、常に前面表示・透明度調整可能
- **参考リポジトリ**: [xai-org/grok-build](https://github.com/xai-org/grok-build)
  - 認証は `auth.x.ai` の OAuth2（ブラウザログイン / Device Code）
  - クレデンシャルは `~/.grok/auth.json` に保存（access_token + refresh_token）
  - `/usage` は CLI 内のスラッシュコマンドでクレジット/使用状況を表示
  - 消費者向け週間使用量は公開 API ドキュメントがなく、Web（grok.com Settings → Usage）または内部エンドポイントを利用する必要がある

---

## 2. 機能要件

| 機能 | 詳細 | 優先度 |
|------|------|--------|
| 複数アカウント対応 | アカウントを追加・削除・切り替え可能。各アカウントごとに独立したトークン管理 | 必須 |
| OAuth2ログイン | `auth.x.ai` ベースの OAuth2（Authorization Code + PKCE 推奨）。ブラウザ起動 or Device Code フロー | 必須 |
| 週間使用量表示 | 使用率（%）、製品別内訳（Chat / Build / Imagine / Voice / API など）、残り量の目安 | 必須 |
| リセット日時表示 | 次の週間リセット日時（ローカルタイムゾーン変換）を常時表示 | 必須 |
| 常に全面（Always on Top） | ウィンドウを常に最前面に固定可能（トグル） | 必須 |
| 透明度変更 | 0〜100% のスライダーまたはプリセットでリアルタイム変更 | 必須 |
| 軽量動作 | メモリ < 50MB 目標、CPU ほぼ0（ポーリング時のみ）、起動高速 | 必須 |
| システムトレイ常駐 | トレイから全操作可能。アイコンに簡易使用率表示 | 推奨 |
| 自動リフレッシュ | 間隔設定可能（デフォルト 5〜15分） | 推奨 |
| テーマ | ダーク/ライト切り替え | 推奨 |

---

## 3. 非機能要件（軽量化重視）

- **バイナリサイズ**: 10MB 以下を目標
- **メモリ使用量**: アイドル時 30MB 以下
- **CPU**: アイドル時ほぼ 0%、更新時のみ短時間
- **更新頻度**: デフォルト 10分間隔（ユーザー設定可能）。バックグラウンドで refresh_token を使って自動更新
- **クロスプラットフォーム**: Windows / macOS / Linux（優先は Windows + macOS）
- **依存最小化**: 外部ランタイム不要（ネイティブ or 組み込み WebView）

---

## 4. 推奨技術スタック（軽量優先）

### 最推奨: Tauri 2.x (Rust + 軽量 WebView)
- バイナリが小さく、Always-on-Top・透明度・システムトレイが容易
- フロントは HTML/CSS/JS（または Svelte / Vue の最小構成）
- バックエンドは Rust で OAuth・HTTP クライアント・トークン管理

### 代替案
| 技術 | メリット | デメリット | 推奨度 |
|------|----------|------------|--------|
| 純 Rust + egui / iced | 完全ネイティブ、さらに軽量 | UI実装コスト高め | 中 |
| Electron | 開発速度速い | 重い（非推奨） | 低 |
| Python + customtkinter / PyQt | 開発速度優先 | 配布時に肥大化しやすい | 低 |

**選定理由**: Grok Build 自体が Rust 製であり、認証周りと親和性が高い。Tauri なら WebView を使いつつネイティブウィンドウ制御が可能。

## 5. アーキテクチャ

```
┌─────────────────────────────────────────────────────┐
│                  Desktop Overlay UI                 │
│  (Always-on-Top / Opacity / Multi-account cards)    │
└──────────────────────┬──────────────────────────────┘
                       │ IPC / Event
┌──────────────────────▼──────────────────────────────┐
│                 Core Engine (Rust)                  │
│  - Account Manager                                  │
│  - OAuth2 Client (PKCE)                             │
│  - Token Store (暗号化ローカルDB or JSON)            │
│  - Usage Poller (バックグラウンドタスク)             │
│  - System Tray                                      │
└──────────────────────┬──────────────────────────────┘
                       │ HTTPS
          ┌────────────┴────────────┐
          │                         │
   auth.x.ai (OAuth)          内部Usage API
   (token取得/更新)            (推定エンドポイント)
```

### コンポーネント責務

| コンポーネント | 責務 |
|----------------|------|
| Account Manager | アカウントの追加・削除・並び替え・現在選択状態管理 |
| OAuth2 Client | Authorization Code + PKCE / Device Code フロー実装 |
| Token Store | access_token / refresh_token の安全な保存・自動更新 |
| Usage Poller | 定期的な使用量取得・エラーハンドリング・バックオフ |
| UI Layer | カード表示・ゲージ・透明度制御・Always-on-Top |

## 6. 認証設計（OAuth2 複数アカウント）

Grok Build の認証ドキュメントに準拠。

### 基本情報
- **エンドポイント**: `https://auth.x.ai`（SpaceXAI OAuth）
- **推奨フロー**: Authorization Code + PKCE  
  （ループバック `http://127.0.0.1:<random_port>/callback`）
- **Device Code フロー**: ヘッドレスやリモート環境用にも対応
- **スコープ例**: `openid profile email offline_access` など（CLI と同様）

### トークン保存
- 各アカウントごとに独立して保存
- 保存先候補:
  - 優先: OS キーチェーン（Windows Credential Manager / macOS Keychain / Linux Secret Service）
  - フォールバック: AES-256 暗号化ローカルファイル（例: `~/.grok-monitor/accounts/<id>.enc`）
- 保存項目:
  - `access_token`
  - `refresh_token`
  - `expires_at`
  - `account_name` / `email` / `sub`
  - `created_at` / `last_used`

### 複数アカウント管理
- アカウントID（email または sub）で一意に識別
- 追加時は新規 OAuth フローを起動
- 既存 Grok CLI 連携（オプション）: `~/.grok/auth.json` を読み込んでインポート可能にする（セキュリティ注意書き必須）

### 再認証
- トークン期限切れや refresh 失敗時は UI 上で「再ログイン」ボタンを表示
- クリックでブラウザを開いて再認証フローを開始

## 7. 使用量取得設計

**重要**: 週間使用量の公式公開 API は現時点で確認できない。

### 想定取得方法
1. OAuth で取得した `access_token` を使い、grok.com の Settings → Usage ページが叩いている内部 REST エンドポイントを利用（要リバースエンジニアリング）
2. レスポンスから以下を抽出:
   - `usage_percentage`（全体使用率）
   - `breakdown`（製品別パーセンテージまたはコスト）
   - `reset_at`（ISO8601 のリセット日時）
   - Extra Usage Credits 残高（存在する場合）

### 実装方針
- 初回はブラウザ開発者ツールで実際のリクエストをキャプチャし、エンドポイントと必要なヘッダーを特定する
- ポーリング間隔は設定可能（デフォルト 10分）
- 失敗時は指数バックオフ + ユーザーへの通知
- レートリミットに配慮（過度なポーリングを禁止）
- エンドポイント変更時は「データ取得不可」と明示的に表示し、アプリ更新を促す

### 抽象化
```rust
// 擬似コード
trait UsageProvider {
    async fn fetch_usage(&self, token: &str) -> Result<UsageData, Error>;
}

struct UsageData {
    percentage: f32,
    breakdown: HashMap<String, f32>,
    reset_at: DateTime<Utc>,
    extra_credits: Option<f64>,
}
```

## 8. UI/UX 設計

### メインウィンドウ
- コンパクトなカード型オーバーレイ
- 各アカウントを縦に並べる、またはタブ/セレクタで切り替え
- 大きな円形ゲージ or 横バーで使用率を視覚化
- リセットまでの残り時間を強調表示（例: `残り 2日 14時間`）
- 製品別ミニ内訳（ホバーまたは展開で詳細）

### コントロール
- 透明度スライダー（ウィンドウ右上 or トレイメニュー）
- Always-on-Top トグル
- 手動更新ボタン
- 自動更新間隔設定
- アカウント追加・削除・並び替え

### システムトレイ
- 右クリックメニューで全機能操作可能
- トレイアイコンに全体使用率を簡易表示（色分け: 緑 → 黄 → 赤）

### テーマ
- ダーク基調（Grok っぽい黒 + アクセントカラー）をデフォルト
- ライトテーマも選択可能

## 9. データ永続化・セキュリティ

| 項目 | 方針 |
|------|------|
| トークン | OSキーチェーン優先 + AES-256フォールバック |
| 設定 | TOML または JSON（更新間隔、透明度、ウィンドウ位置・サイズなど） |
| ログ | ローカルのみ。センシティブ情報は出力しない |
| 権限 | 最小権限の原則。ネットワークは HTTPS のみ |

---

## 10. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 内部 Usage API の変更・廃止 | データ取得不能 | 抽象レイヤーを用意。取得失敗時は明示的に「要更新」表示。コミュニティでエンドポイント情報を共有 |
| ToS 違反の可能性 | アカウント制限 | 個人利用・読み取り専用を明記。過度なリクエストを避ける |
| トークン漏洩 | アカウント乗っ取り | 暗号化 + 最小権限 + ソース公開時はセキュリティ注意書きを徹底 |
| マルチアカウントのセッション管理 | 予期せぬログアウト | 各アカウント独立トークン。同時ログイン数制限は xAI 側に依存 |

## 12. 今後の調査ポイント

1. Grok Build の `/usage` が返す実際の JSON 構造を CLI で確認する
2. grok.com の Settings → Usage ページを DevTools でキャプチャし、正式なエンドポイント・ヘッダー・レスポンス形式を特定する
3. `auth.x.ai` の正確な `client_id` / `redirect_uri` / 必要なスコープを CLI の挙動またはネットワークキャプチャから取得する
4. 週間リセットの正確なタイミング（UTC かローカルか、曜日・時刻）を複数アカウントで確認する
